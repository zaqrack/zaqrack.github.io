Application dispatching

HPE TAS offers multiple approaches for solutions needing multiple applications. HPE TAS multi-application and application dispatching mechanism provides a visual overview of the possibilities.

The mechanism for application dispatch is the following:

  • The front-end loadbalancers load-balance traffic between multiple HPE TAS SEE instances.
  • The SEE instance routes the network message to the service. (By declaring use of a provider and configuring its connectivity attributes).
  • If the service happens to be the SCXML orchestrator, then an SCXML script is executed to decide the application service chain to be executed. If the service-component happens to be a service (deployed without orchestrator), then it is executed.

Determining which application or service-chain to dispatch:

  • Invoke java procedures (stored in a .bsh file) from the SCXML script can be used to check the underlying network protocol, extract message-elements/fields from network messages to take the orchestration decision, that is, deciding which service-chain to execute, or
  • SCXML script could unconditionally pass the callStart event to a dedicated routing SCIF or MCIF service (written by a developer) which then uses the Java abstract network APIs to determine the underlying network protocol and extract message-elements/fields from the network message, and returns a parameter to the Orchestration layer (SCXML script) which determines the service-chain to execute.

The message-elements/fields could be Global Title, SSN, Application Context Name, Service-Key for INAP/CAP/MAP protocols, or could be SIP method-name, R-Uri, and specific header values for the SIP protocol.
